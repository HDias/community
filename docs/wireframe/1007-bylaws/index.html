UX Notes

  • ① Bylaws lives as a top-level sidebar item; members land directly on the single authoritative document — no list to navigate first.
  • ② A prominent pending-acknowledgement banner is the safe pattern's main driver of completion: it appears only while the member hasn't confirmed the current version.
  • ③ The changes summary is surfaced in a bordered callout above the text so members see what's new before reading the full document.
  • ④ The acknowledge action is a single explicit checkbox + button; copy notes that name and timestamp are recorded, reinforcing the audit trail (acknowledged_at).

  • ① Two version pickers (From → To) let an admin diff any pair, not just consecutive versions — useful for auditing changes over time.
  • ② Previous and Current sit side by side; the current column is subtly distinguished with an #eee header fill rather than color.

UX Notes

  • ① A 3-step stepper (Edit → Summarize → Review) makes publishing a deliberate, high-stakes action rather than an inline save, reducing accidental versions.
  • ② Step 1's content lives in a textarea (only admins reach this flow), carried forward for editing here.
  • ③ A dedicated changes-summary field is required mid-flow, guaranteeing every version ships with the summary members receive.
  • ④ The review step spells out the consequence explicitly — new version number, notifies all 128 members by email + in-app, and creates pending acknowledgements — so the admin confirms with full awareness.
